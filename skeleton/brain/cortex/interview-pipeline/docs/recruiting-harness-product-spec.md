# HireBase Harness 产品化规范

> 目标：把招聘从零散聊天、临时约面和事后回忆，变成一条可执行、可复盘、可交接的工作流。系统维护内部资料，候选人只接收必要通知。

---

## 0. 产品定位

**一句话**：HireBase Harness 是一个以飞书多维表格为主控台的 AI 招聘中控台。

它把招聘平台、飞书多维表格、飞书文档、会议系统、简历解析、候选人评分、面试预约、主持稿生成、会议录制转写、面试复盘、测试题跟进和二面流转，全部收进同一张候选人表里。用户主要通过多维表格查看信息、筛选候选人、确认状态和推进流程；AI 负责重复劳动、资料归档和过程推进。

核心原则：

1. **先打通工具，再进入招聘流程**：招聘平台 CLI、飞书 CLI、会议 CLI 是三条主干链路，任何自动化都建立在它们可用的前提上。
2. **先配置环境，再处理候选人**：Node、Python、CLI、token、环境变量、表格字段、会议录制权限都要在流程启动前检查。
3. **飞书多维表格是单一真相源**：简历、评分、会议、主持稿、转写、复盘、测试题、状态全部回到同一条候选人记录。
4. **AI 做执行，人做决定**：AI 可以推荐、准备、记录、归纳，但约谁、过不过、何时进入二面必须等用户明确指令。

### 0.1 主控台：飞书多维表格

HireBase Harness 的核心不是 README、聊天窗口或单个飞书文档，而是飞书多维表格。用户应当能只打开这张表，就看到每个候选人的完整生命周期和当前待办。

多维表格承担四件事：

1. **信息总览**：候选人是谁、简历在哪里、AI 怎么评、风险是什么、现在卡在哪一步。
2. **状态流转**：待筛选、一面待约、一面已约、一面已完成、测试题、二面、通过/不通过都由表格状态表达。
3. **动作入口**：会议链接、主持稿链接、录音转写、复盘、测试题材料都挂回候选人同一行。
4. **团队协作**：用户、面试官和 AI 都围绕同一行记录协作，避免散落在聊天、文档和会议历史里。

其他载体都是多维表格的附属对象：

| 区域 | 作用 | 谁看 | 写什么 |
|---|---|---|---|
| 飞书多维表格 | 主控台和单一真相源 | 用户 / 招聘团队 / AI | 简历、评分、风险、状态、会议链接、面试稿链接、录音转写、复盘、测试题状态 |
| 飞书面试文档 | 表格某一行的深度工作页 | 面试官 / AI | 开场脚本、候选人速览、核心问题、红旗前置、测试题承接、反问预备回答、面试官备忘 |
| 候选人沟通窗口 | 表格状态触发的对外动作 | 候选人 / 招聘方 | 收到简历确认、询问是否有空、发送会议链接、必要的测试题说明 |
| 会议系统 | 表格中某一轮面试的执行载体 | 面试官 / 候选人 | 会议链接、录制、转写、参会记录 |

边界：

- 内部判断首先写在飞书多维表格；飞书面试文档只承接当场面试需要展开的细节。
- 候选人沟通只给必要信息，不暴露内部轮次、评级、排序、薪资策略、淘汰逻辑。
- 飞书面试文档不是候选人材料，是从多维表格行派生出来的面试官操作台。
- 任何自动化动作完成后，都必须回写多维表格；没有回写就视为流程未完成。

### 0.2 新用户首次启动流程

新用户第一次使用时，不应该直接进入“同步简历”或“约面”。系统必须先把环境和配置跑通，让用户知道自己还缺什么。

首次启动顺序：

1. **环境体检**：检查 Node、Python、包管理器、招聘平台 CLI、飞书 CLI、会议平台 CLI。
2. **登录态体检**：确认三类平台都已登录或授权；只展示“已配置/未配置”，不展示 token、cookie、密钥。
3. **权限体检**：确认能读候选人消息、能读写飞书多维表格、能创建飞书文档、能创建会议、能开启录制和转写。
4. **项目初始化**：引导用户填写岗位画像、公司介绍、面试轮次、测试题规则、消息模板。
5. **多维表格初始化**：新建或绑定飞书多维表格，缺字段就创建或提示迁移。
6. **试运行**：用一份样例简历跑通“解析 → 入表 → 评分 → 生成面试稿”的最小闭环。
7. **进入正式流程**：只有试运行通过后，才开启真实简历监听和自动通知。

这一步是产品化的关键：它保证其他人拿到 Harness 后，不需要理解历史项目背景，也能按引导把自己的招聘流程跑起来。

新项目应从 [`recruiting-project-config-template.md`](recruiting-project-config-template.md) 复制一份私有配置，再进入正式流程。

## 1. 前置能力与环境检查

### 1.1 必须打通的三类 CLI

| 能力 | 目的 | 必须验证的动作 |
|---|---|---|
| 招聘平台 CLI | 接收候选人消息、监听简历、自动回复、发送面试通知、监听确认 | 能读取候选人消息；能识别简历附件；能发送文本通知；能记录候选人回复 |
| 飞书 CLI | 读写多维表格、写文档、发通知、建日程、管理附件 | 能读写多维表格；能创建/更新文档；能上传附件；能创建日程或会议 |
| 会议平台 CLI | 创建会议、打开云录制/云录音/转写、拉取会后转写 | 能创建会议；能配置云录制和转写；能查询结束会议；能拉取转写文本 |

> 产品化要求：启动招聘项目前，系统必须展示这三类 CLI 的连通性结果。未打通的能力标红，并提示用户先授权或配置，不直接进入自动化流程。

常见实现映射：

| 通用能力 | 可选实现 |
|---|---|
| 招聘平台 CLI | Boss 直聘、猎聘、拉勾、LinkedIn、邮件投递入口 |
| 协作平台适配 | 默认飞书；其他协作平台只能作为适配层，不能改变“表格主控台”的产品形态 |
| 会议平台 CLI | 腾讯会议、飞书会议、Zoom、Google Meet、Teams |

### 1.2 运行环境检查

系统启动时先做环境体检，默认展示给用户：

| 检查项 | 用途 | 失败时处理 |
|---|---|---|
| Node.js / pnpm | 运行解析器、监听器、自动化脚本 | 给出安装或切换版本流程 |
| Python / 文档解析依赖 | 解析 PDF、docx、转写和表格导入 | 给出安装依赖流程 |
| 招聘 / 飞书 / 会议 CLI 路径 | 确认命令可调用 | 提示安装、登录或刷新 PATH |
| token / 登录态 | 访问招聘消息、表格、会议 | 引导重新登录，不输出敏感 token |
| 环境变量 | 控制代理、工作目录、项目 ID | 展示变量名和是否存在，不展示敏感值 |
| 多维表格字段结构 | 确认候选人表可写 | 缺字段则提示创建或迁移 |
| 会议录制权限 | 确保面试后有转写 | 未开启则禁止自动约面，提示先授权 |

### 1.3 环境配置流程

1. **检测本机环境**：输出 Node、Python、CLI、登录态、环境变量、飞书多维表格字段、会议权限的状态。
2. **生成缺口清单**：把缺失项分成「必须修复」「可稍后补」「人工确认」三类。
3. **逐项引导配置**：例如安装 Node、登录协作平台、登录会议平台、配置招聘平台授权、补充表格字段。
4. **二次验证**：每修完一项立即重新检测。
5. **锁定项目配置**：通过后写入项目级配置，后续流程只读配置，不在任务中临时猜。

## 2. 项目初始化

招聘开始前，系统引导用户提供基础材料：

1. **岗位画像**：想找什么人、岗位职责、硬性要求、加分项、淘汰项、薪资范围、地点要求、到岗时间。
2. **公司介绍**：公司做什么、团队规模、融资/商业化状态、工作方式、候选人常问问题的标准回答。
3. **面试流程规则**：是否一面/二面、每轮谁面、每轮题型、是否有测试题、通过标准。
4. **飞书多维表格结构**：新建或绑定候选人表，确认字段、视图和状态枚举完整。
5. **消息模板**：简历收到回复、时间确认、会议确认、测试题通知等；候选人外发不写内部轮次和评价。

初始化完成后，系统生成一个招聘项目配置：

- `job_profile`：岗位画像
- `company_profile`：公司介绍
- `candidate_base`：飞书多维表格主控台
- `interview_rounds`：面试轮次规则
- `message_templates`：候选人沟通模板
- `tool_status`：三类 CLI 与环境检查结果

### 2.1 通用规范与项目私有配置的边界

通用 Harness 只能沉淀流程、字段、状态机、检查项和模板，不沉淀任何实例身份。

| 内容 | 放在哪里 | 能否随 Skill 导出 |
|---|---|---|
| 招聘前/中/后流程 | 通用规范 | 可以 |
| 环境体检清单 | 通用规范 | 可以 |
| 飞书多维表格字段模板 | 通用规范 | 可以 |
| 状态机 | 通用规范 | 可以 |
| 消息模板 | 通用规范 | 可以，需用变量占位 |
| 公司名称、岗位名称、薪资、地点 | 项目配置 | 不直接写死在通用 Skill |
| 平台账号、资源标识符、会议链接 | 项目私有配置 | 不导出 |
| 候选人姓名、简历、录音、复盘 | 项目数据 | 不导出 |

对外可复用的 Skill 应该只要求新用户填写配置，而不是继承某个历史项目的个人信息。

### 2.2 飞书多维表格字段分区

飞书多维表格要按“基础信息、内部判断、面试轮次、测试题、外联动作”分区，不要把所有内容混在一个备注里。字段命名要服务于用户扫表：打开表格后，用户应当能直接筛出“今天要面的人”“一面后待复盘的人”“测试题卡住的人”“二面待约的人”。

| 分区 | 字段 | 用途 |
|---|---|---|
| 基础信息 | 候选人、来源、简历、工作年限、城市、期望薪资 | 识别候选人和现实条件 |
| AI 评估 | AI 摘要、评分、星级、风险、推荐动作 | 帮用户挑人 |
| 一面 | 一面状态、时间、会议链接、主持稿、录音/转写、复盘、结果 | 一面闭环 |
| 测试题 | 测试题状态、提交材料、评估结论 | 承接作业或实操题 |
| 二面 | 二面状态、时间、会议链接、主持稿、录音/转写、复盘、结果 | 二面闭环 |
| 外联 | 最近一次候选人通知、候选人回复、是否确认参会 | 只记录对外触达状态 |

候选人表的状态字段是流程开关。AI 可以建议，是否推进由用户决定。

### 2.3 推荐视图

为了让用户通过多维表格管理流程，至少需要这些视图：

| 视图 | 筛选/排序 | 用途 |
|---|---|---|
| 全部候选人 | 按入表时间倒序 | 总览候选人池 |
| 待筛选 | 当前状态=待筛选，按星级/评分倒序 | 集中看新简历 |
| 待约面 | 当前状态包含待约，按评分倒序 | 给定时间窗口后从这里选人 |
| 今日面试 | 面试日期=今天，按时间升序 | 面试当天排程 |
| 面后待复盘 | 状态=已完成且复盘为空 | 防止面完不归档 |
| 测试题跟进 | 测试题状态不为空且未终结 | 跟踪作业进度 |
| 二面池 | 当前状态=二面待约/二面已约 | 管理后续轮次 |
| 已终结 | 当前状态=通过/不通过/放弃 | 归档，不参与自动推荐 |

AI 每次批量动作结束后，要能说明“更新了哪些记录、哪些视图会发生变化、还有哪些记录需要人工确认”。

## 3. 招聘前

### 3.1 接收岗位需求

用户提供文档或口述信息后，AI 需要整理成结构化岗位画像：

| 模块 | 内容 |
|---|---|
| 必须具备 | 年限、技术栈、行业经验、地点、薪资、语言、学历等硬条件 |
| 加分项 | AI/LLM、自动化、会议/语音/ASR、创业团队经验等 |
| 红旗项 | 投错岗、薪资倒挂、异地不可来、管理岗降级风险、履历空窗 |
| 评估维度 | 技术匹配、业务匹配、学习能力、稳定性、沟通表达、薪资地点可行性 |

### 3.2 发布招聘后监听简历

招聘信息发布后，系统进入监听状态：

1. 监听候选人新消息和简历附件。
2. 识别是否为新候选人或重复候选人。
3. 一旦收到简历，自动回复：

```text
已收到你的简历，我们会先做岗位匹配评估。若匹配度合适，会继续与你沟通面试安排。
```

4. 将简历下载并解析，写入候选人表。
5. 对候选人评级和评分。

候选人收到的回复只确认“已收到”，不要提内部评分、后续轮次、候选人排序。

### 3.3 候选人入表与评级

一位候选人一条记录。最低字段：

| 字段 | 说明 |
|---|---|
| 候选人 | 姓名或平台昵称 |
| 来源 | 招聘平台 / 内推 / 邮件 / 协作平台转发 |
| 简历 | 原始附件 |
| AI 摘要 | 背景一句话 |
| 评分 | 数值分，便于排序 |
| 星级 | 高 / 中 / 低 |
| 风险 | 异地、薪资倒挂、投错岗、空窗等 |
| 推荐动作 | 优先约面 / 可约 / 暂缓 / 不建议 |
| 当前状态 | 待筛选 |

评分建议采用 100 分制：

| 维度 | 权重 |
|---|---:|
| 岗位硬匹配 | 30 |
| 关键经验匹配 | 25 |
| AI / 自动化 / 工具链能力 | 20 |
| 薪资地点可行性 | 15 |
| 沟通和稳定性信号 | 10 |

## 4. 招聘中

招聘中分为预约面试、准备面试稿、面试进行中三段。

### 4.1 预约面试

预约面试必须等待用户明确指令，例如：

```text
今天下午 3 点到 6 点可以面试 6 位候选人，每位 30 分钟。
```

收到指令后，系统执行：

1. 从候选人表中筛选状态为「待筛选 / 一面待约」且未被淘汰的人。
2. 按评分、星级、风险、用户偏好排序。
3. 选择最高优先级的 N 位候选人。
4. 生成时间槽，例如 15:00、15:30、16:00、16:30、17:00、17:30。
5. 使用飞书日历或会议平台创建会议。
6. 内部会议名称写清楚，方便团队检索；候选人外发文案不暴露内部轮次和评级。
7. 会议描述写清楚面试官、候选人、预计时长、准备事项。
8. **自动打开音频转写、云录音或云录制**。如果工具不支持自动打开，必须在结果中标红提醒用户手动打开。
9. 将会议链接、时间、轮次、面试官写回候选人表。
10. 通过招聘平台或候选人可触达渠道发送极简时间确认。
11. 监听候选人回复。
12. 候选人确认收到后，把状态更新为「一面已约」或「二面已约」。

第一次询问是否有空：

```text
你好，请问你 {日期} {时间段} 是否方便做一次线上沟通？预计 {时长} 分钟左右。
如果这个时间不方便，也可以告诉我你方便的时间段。
```

候选人确认后再发会议链接：

```text
好的，那我们约在：
时间：{日期} {开始时间}-{结束时间}
会议：{会议链接}

到时麻烦提前 3 分钟进入会议，保持麦克风和网络可用。谢谢。
```

不要在候选人外发消息里写：

- 第几轮面试
- 内部评级或优先级
- 薪资策略
- “你是我们最匹配/最强候选人”
- 测试题真实考察意图
- “通过/不通过”的暗示

### 4.2 准备面试稿

会议约好后，系统立即生成面试官内部文档。每位候选人每一轮一份，默认写入飞书文档空间，并把链接回填候选人表。

面试文档结构见 [`feishu-interview-doc-template.md`](feishu-interview-doc-template.md)，核心结构：

1. **文档头部信息**：面试时间、候选人、岗位、内部评级、本轮目标。
2. **开场介绍公司**：用统一公司口径，能直接照读。
3. **候选人速览**：核心经历、匹配点、风险点、本轮必须验证。
4. **简历追问**：每题从简历事实引出。
5. **红旗前置**：薪资、地点、到岗、作业未交等必须前置。
6. **测试题承接**：如果有作业，逐条写内部判断和追问。
7. **反问预备回答**：团队、产品、技术、工作节奏的标准口径。
8. **结束语**：克制，不承诺结果。
9. **面试官备忘**：KILL/PASS 信号、本轮决策点、下一步建议。

追问话术铁律：

```text
我从你简历里看到 X，所以想问你 Y。
```

### 4.3 面试进行中

面试由用户实际进行，AI 不抢决策权。

面试前检查：

- 会议链接可打开
- 云录制 / 云录音已开启
- 实时转写已开启
- 面试稿链接可打开
- 候选人已确认参会

面试时用户使用准备好的逐字稿。AI 的职责是保证材料齐、链路通、录音转写不会丢。

面试时不要把面试文档整篇共享给候选人。它是面试官内部脚本。

---

## 5. 招聘后

### 5.1 拉取转写并回填

面试结束后，系统监听会议结束状态：

1. 查询已结束会议。
2. 找到候选人对应会议。
3. 拉取云录制、音频或文字转写。
4. 校验转写中的候选人身份与表格记录一致。
5. 把转写文档或附件写回候选人表。
6. 若转写缺失，标记为「待补录音 / 待手动上传」。

### 5.2 面试评级与复盘

AI 基于转写生成本轮面试复盘：

| 模块 | 内容 |
|---|---|
| 总体结论 | 推荐推进 / 待定 / 不建议 |
| 面试表现 | 表达、逻辑、真实度、主动性 |
| 岗位匹配 | 与岗位画像逐项对齐 |
| 核心亮点 | 面试中出现的强信号 |
| 主要风险 | 红旗是否被证实 |
| 下一步建议 | 测试题 / 二面 / 不通过 / 待用户判断 |

同时写入分数：

- 本轮面试评分
- 是否建议进入下一轮
- 是否需要测试题
- 测试题状态

### 5.3 测试题状态

如果流程包含测试题，表格中必须有独立状态：

| 状态 | 含义 |
|---|---|
| 未发放 | 还没有发题 |
| 测试中 | 已发题，等待提交 |
| 已提交待评估 | 候选人已交付 |
| 通过 | 测试题有效 |
| 不通过 | 测试题不达标 |
| 放弃 | 候选人未继续 |

AI 可以记录测试题状态变化，但进入下一轮必须等待用户明确确认。

### 5.4 二面流转

一面结束后，系统只做建议，不自动进入二面。

进入二面需要满足：

1. 一面转写已回填。
2. 一面复盘已生成。
3. 测试题如果存在，状态已明确。
4. 用户明确说进入二面、约二面或推进下一轮。

二面流程与一面一致，但候选人外发话术仍然保持克制：

预约下一次沟通 → 创建会议并开启录制转写 → 写内部主持稿 → 向候选人确认时间并发送链接 → 等确认 → 面试 → 拉转写 → 写复盘 → 等用户终态判断。

---

## 6. 状态机

推荐候选人主状态：

```text
待筛选
→ 一面待约
→ 一面已约
→ 一面已完成
→ 测试题未发放 / 测试中 / 已提交待评估 / 测试题通过 / 测试题不通过
→ 二面待约
→ 二面已约
→ 二面已完成
→ 通过 / 不通过 / 放弃
```

状态规则：

- AI 可以建议状态变化。
- AI 可以在用户授权后执行状态写入。
- 涉及淘汰、通过、进入二面、发 offer 的动作必须等用户明确指令。
- 会议状态和候选人主状态要分开：候选人可能「一面已约」，但会议可能「候选人未确认」。

## 7. 自动化巡检与手动刷新

HireBase Harness 不做高频轮询。默认只保留一个轻量的每日自动巡检：**每天下午 4 点**检查一次飞书多维表格和相关外部状态。其他时间如需更新，由系统提醒用户手动触发刷新。

默认机制：

| 机制 | 触发方式 | 默认频率 | 动作 |
|---|---|---:|---|
| 每日多维表格巡检 | 自动 | 每天 16:00 | 扫描状态、缺失字段、到期任务、待补链接、待复盘记录 |
| 手动刷新简历 | 用户指令 | 按需 | 拉取招聘平台新简历、入表、评分 |
| 手动刷新候选人回复 | 用户指令 | 按需 | 检查候选人是否确认参会，回写表格 |
| 手动刷新会议产物 | 用户指令 | 按需 | 拉取录制/转写、回填表格、生成复盘 |
| 手动刷新测试题 | 用户指令 | 按需 | 检查提交状态、更新测试题字段 |

每日 16:00 巡检负责发现问题和提醒，不负责替用户做高风险决策。巡检发现需要即时更新的外部信息时，默认提示用户执行手动刷新，例如：

```text
发现 3 条候选人记录可能需要更新：2 条缺候选人回复确认，1 条缺会议转写。
是否现在执行手动刷新？
```

每日多维表格巡检至少检查：

| 检查对象 | 发现问题 | 动作 |
|---|---|---|
| 待筛选记录 | 有简历但无 AI 评估 | 解析简历并写评分 |
| 待约面记录 | 用户已给时间窗但未排会议 | 进入候选人排序和约面队列 |
| 已约记录 | 缺会议链接、候选人未确认、会议时间与表格不一致 | 补链接、监听回复、同步会议时间 |
| 今日面试 | 缺主持稿、缺录制/转写确认 | 生成文档、提醒开启录制/转写 |
| 已完成面试 | 缺转写、缺复盘、缺结果建议 | 拉会议产物、生成复盘、写回表格 |
| 测试题记录 | 截止临近、未提交、已提交未评估 | 更新测试题状态并提示用户 |
| 二面池 | 一面通过但二面未约 | 标记“二面待约”，等待用户明确时间指令 |

禁止让自动巡检直接做这些事：

- 擅自把候选人改成通过/不通过。
- 擅自进入二面或发测试题。
- 擅自给候选人发送带内部判断的信息。
- 在缺少录制/转写权限时静默约面。

自动巡检和手动刷新都必须有游标：

- `last_recruiting_message_at`
- `last_resume_processed_at`
- `last_table_updated_at`
- `last_meeting_checked_at`
- `last_test_submission_checked_at`
- `last_daily_audit_at`

所有游标都应项目级保存，防止重复处理和漏处理。

每次自动巡检或手动刷新结束都要写一条自动化日志，至少包含：

| 字段 | 内容 |
|---|---|
| run_id | 本次巡检或刷新 ID |
| started_at / finished_at | 开始和结束时间 |
| source | 多维表格 / 招聘平台 / 会议平台 / 测试题 |
| scanned_count | 扫描记录数 |
| updated_count | 更新记录数 |
| skipped_count | 跳过记录数 |
| error_count | 失败记录数 |
| next_cursor | 下一次游标 |

日志可以写入项目本地文件，也可以写入飞书多维表格的“自动化日志”子表。只要进入真实自动化，必须能追溯“为什么这条记录被改了”。

## 8. 输出形态

这套 Harness 可以输出成三种形态：

1. **Skill**：适合当前 Codex/Cinder 使用。触发后按本规范执行招聘流程。
2. **CLI 工作流**：适合本地自动化，提供 `init`、`check-env`、`sync-resumes`、`schedule`、`write-docs`、`pull-transcripts`、`review` 等命令。
3. **产品界面**：适合 SaaS 化，提供项目配置、候选人看板、环境体检、自动化日志、面试队列和复盘中心。

建议命令：

```text
recruit-harness init
recruit-harness check-env
recruit-harness sync-resumes
recruit-harness rank-candidates
recruit-harness schedule --from 15:00 --to 18:00 --count 6 --duration 30
recruit-harness write-interview-docs
recruit-harness pull-transcripts
recruit-harness review-interviews
```

## 9. 当前 Skill 化要求

如果落成 Skill，`SKILL.md` 应表达这些触发和边界：

触发词：

- 招聘
- 招聘平台
- 简历
- 候选人
- 面试
- 约面
- 面试稿
- 会议平台
- 测试题
- 二面
- 招聘 Harness

Skill 启动顺序：

1. 读取本产品化规范。
2. 读取项目配置模板或当前项目私有配置。
3. 读取面试文档模板。
4. 如果是已有项目，读取候选人表上下文和在线面试文档现状。
5. 先检查招聘平台 / 协作平台 / 会议平台 CLI 和环境变量。
6. 根据用户请求落到招聘前、招聘中、招聘后其中一个阶段。
7. 执行前确认是否需要用户明确授权。

硬边界：

- 不读取或输出敏感 token。
- 不绕过 CLI 白名单和授权。
- 不擅自淘汰候选人。
- 不擅自进入二面。
- 不在会议未开启录制/转写时静默约面。
- 不把 A 候选人的简历、录音、复盘写到 B 候选人记录。
- 不用本地旧稿整篇覆盖在线面试文档；修改已有文档前必须先读取在线现状。
- 不向候选人暴露内部轮次、评级、排序、薪资策略、淘汰逻辑。

---

## 10. 一句话总览

**招聘前**：配置岗位和工具，监听简历，自动回复，入表评分。

**招聘中**：等用户给时间窗口，按优先级选人，创建会议并开启录制转写，用极简话术向候选人确认时间和会议链接，确认后写内部面试文档。

**招聘后**：拉转写，写复盘，更新测试题和轮次建议，等用户明确指令后进入二面或终态。
